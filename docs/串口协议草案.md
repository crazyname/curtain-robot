# 智能窗帘串口协议草案

## 1. 文档目的

本文档用于梳理智能窗帘项目当前已经出现的串口命令，并提出一份后续统一 STM32 固件、Python 控制程序、微信小程序三端通信格式的目标协议草案。

本文档只描述协议现状和后续设计方向，不表示目标协议已经全部实现。当前代码仍以已有命令为主，后续需要按迁移计划逐步补齐。公开入口、总体能力边界和当前开发优先级由根目录 `README.md` 汇总。

### 当前实现状态

- STM32 侧已基础支持 `STOP`。
- STM32 侧已基础支持 `STATUS`。
- Python 控制程序已支持最小 `STOP` / `STATUS` 调用及 `STATE:<state>;CAL:0` 状态解析，并以模拟测试覆盖新响应等待和超时边界。
- 微信小程序尚未接入统一 `STOP` / `STATUS` 调用。
- Python 与 STM32 的实物串口联调、完整校准状态和电机动作验证尚未完成。
- 当前尚未完成三端统一协议下的完整跨端闭环。

## 2. 当前已有命令梳理

当前已有命令主要来自现有 STM32 串口控制逻辑和上位控制思路，命令格式较短，语义依赖代码上下文。

| 当前命令 | 当前含义 | 备注 |
| --- | --- | --- |
| `ON` | 打开窗帘 | 可作为后续 `OPEN` 的旧命令兼容入口 |
| `OFF` | 关闭窗帘 | 可作为后续 `CLOSE` 的旧命令兼容入口 |
| `C` | 校准或记录行程相关命令 | 当前语义不够清晰，需要和 `CAL:*` 统一 |
| `CE` | 校准结束或校准扩展相关命令 | 需要和 `C` / `CAL` 命令重新统一定义 |
| `S...T` | 带起止标记的数据命令 | 具体含义依赖中间数据内容，后续建议收敛为明确字段 |

## 3. 当前三端命令使用情况

当前项目存在 STM32 固件、Python 控制程序、微信小程序三个控制入口或潜在控制入口，但命令命名和能力边界尚未统一。

| 端侧 | 当前情况 | 风险 |
| --- | --- | --- |
| STM32 固件 | 已存在 `ON`、`OFF`、`C`、`S...T` 等短命令处理思路，已基础支持 `STOP` / `STATUS` | 状态字段、校准状态和跨端协同仍不完整 |
| Python 控制程序 | 已接入 `STOP` / `STATUS` 最小调用、状态解析及模拟超时处理 | 真实串口、电机动作与后续扩展字段仍需联调验证 |
| 微信小程序 | 后续需要作为用户侧控制入口 | 若不统一协议，容易出现小程序、Python、STM32 三端各发各的命令 |

## 4. 当前问题

当前串口协议主要存在以下问题：

1. 三端命令不统一，STM32、Python、小程序之间缺少同一套协议定义。
2. `C` / `CE` 语义不清，难以判断是校准开始、校准结束，还是其他校准阶段。
3. STM32 与 Python 已有 `STOP` 最小路径，但微信小程序和真实硬件动作验证尚未接入完成。
4. STM32 与 Python 已有 `STATUS` 最小返回/解析路径，但真实串口联调、状态字段完善和跨端统一尚未完成。
5. 缺少位置百分比命令，无法表达移动到 0% 到 100% 中间位置。
6. 缺少统一返回值和错误码，调试时难以区分成功、参数错误、忙碌、未校准等状态。

## 5. 目标协议设计

目标协议建议使用可读性更强的英文命令，并保留简单串口文本格式，方便 STM32、Python、小程序共同实现。

> 注意：本节是目标协议设计，不代表当前已经实现。

| 目标命令 | 目标含义 | 当前实现状态 |
| --- | --- | --- |
| `OPEN` | 打开窗帘 | 目标命令，后续由 `ON` 兼容迁移 |
| `CLOSE` | 关闭窗帘 | 目标命令，后续由 `OFF` 兼容迁移 |
| `STOP` | 立即停止当前运动 | STM32 与 Python 已基础实现，小程序和实物验证待完成 |
| `STATUS` | 查询当前窗帘状态、位置和校准状态 | STM32 与 Python 已基础实现最小返回/解析，扩展字段和实物验证待完成 |
| `SET:50` | 移动到 50% 位置，数值范围建议为 `0` 到 `100` | 尚未实现位置百分比控制 |
| `CAL:START` | 开始校准或开始记录完整行程 | 需要统一迁移的目标命令，需替代或兼容现有 `C` |
| `CAL:END` | 结束校准并保存校准结果 | 需要统一迁移的目标命令，需和现有 `CE` 语义统一 |

目标命令建议约定：

1. 命令使用大写英文。
2. 无参数命令直接发送命令名，例如 `OPEN`、`STOP`。
3. 有参数命令使用冒号分隔，例如 `SET:50`、`CAL:START`。
4. 后续如需扩展，可继续使用 `KEY:VALUE` 格式，避免重新发明多套格式。

## 6. 兼容旧命令方案

为避免一次性改动 STM32、Python、小程序三端，建议先在协议层明确旧命令兼容关系，再逐步迁移实现。

| 旧命令 | 兼容到目标命令 | 处理建议 |
| --- | --- | --- |
| `ON` | `OPEN` | 保留旧命令入口，同时新代码优先发送 `OPEN` |
| `OFF` | `CLOSE` | 保留旧命令入口，同时新代码优先发送 `CLOSE` |
| `C` | `CAL:START` 或校准流程入口 | 需要先确认当前代码语义，再映射到明确的校准阶段 |
| `CE` | `CAL:END` 或校准结束动作 | 需要和 `C` / `CAL` 命令统一，避免同一阶段多个名字 |
| `S...T` | 待拆分为明确命令或参数格式 | 不建议继续作为长期主协议格式 |

兼容策略建议：

1. STM32 先同时识别旧命令和目标命令。
2. Python 和小程序逐步改为只发送目标命令。
3. 等三端稳定后，再决定是否移除旧命令入口。
4. `C` / `CE` 必须先完成语义确认，再决定最终映射关系。

## 7. 返回值与状态格式草案

当前项目尚未完成跨端统一返回值。STM32 已具备基础状态返回，Python 已可解析该最小格式并区分本次响应等待超时；后续仍需完善字段、开展实物联调并接入小程序。

### 7.1 通用返回值草案

| 返回值 | 含义 |
| --- | --- |
| `OK` | 命令执行成功或已接受 |
| `ERR:UNKNOWN_CMD` | 未知命令 |
| `ERR:BAD_PARAM` | 参数错误，例如 `SET:150` |
| `ERR:BUSY` | 当前正在运动或校准，暂不能执行 |
| `ERR:NOT_CALIBRATED` | 尚未完成校准，不能执行位置控制 |
| `ERR:HW_FAULT` | 电机、传感器或驱动异常 |

### 7.2 状态返回格式草案

当前 STM32 的基础状态返回可使用一行文本：

```text
STATE:<state>;CAL:0
```

其中 `CAL:0` 当前是基础/保守字段输出，不代表完整校准状态持久化已经完成。

`STATUS` 的后续目标返回格式可以扩展为：

```text
STATE:IDLE;POS:50;CAL:1
```

字段含义建议如下：

| 字段 | 示例 | 含义 |
| --- | --- | --- |
| `STATE` | `IDLE` / `OPENING` / `CLOSING` / `STOPPED` / `CALIBRATING` | 当前运行状态 |
| `POS` | `0` 到 `100` | 当前估算位置百分比 |
| `CAL` | `0` / `1` | 是否已经完成校准 |

除 STM32 当前已具备、且 Python 已完成最小解析接入的 `STATE:<state>;CAL:0` 返回外，以上完整返回值和扩展状态格式仍是草案，需要根据实际内存、计时、校准状态和跨端解析情况继续调整。

## 8. 后续迁移计划

在公开文档结构完成精简后，建议按以下顺序迁移，避免三端同时大改导致问题难以定位：

1. 基于 Python 已实现的 `STOP` / `STATUS` 最小调用和解析路径，完成与 STM32 的真实串口联调及电机动作验证。
2. 在实物串口闭环可验证后，完善 STM32 状态字段与校准状态，使基础 `STATUS` 返回逐步覆盖实际需求。
3. 再实现 `SET:x` 位置控制，基于校准行程或运行时间估算中间位置。
4. 最后统一小程序命令，让用户侧只发送目标协议命令。

## 9. 下一步开发建议

Python 串口客户端已完成 STM32 `STOP` / `STATUS` 最小代码接入与模拟测试。在此基础上再推进以下事项：

1. 使用真实 STM32 串口验证 Python 的 `STOP` / `STATUS` 发送、最小状态返回等待/解析及超时结果；保留 `CAL:0` 仅为基础占位的边界。
2. 确认 `C` 和 `CE` 在当前固件/旧小程序中的真实语义，决定后续校准命令映射。
3. 完善已有基础 `STATUS` 返回的状态字段和校准状态，再扩展位置百分比。
4. 在基础串口闭环稳定后，再统一小程序目标命令和状态显示。

当前只能将 `STOP` 和 `STATUS` 描述为 STM32 与 Python 的最小代码路径已具备，且 Python 已完成模拟测试；真实串口、电机动作、小程序、`SET:x` 位置控制、完整校准状态和整机跨端闭环仍不得描述为已经完成。
