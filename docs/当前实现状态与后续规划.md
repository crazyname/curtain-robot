# 智能窗帘项目当前实现状态与后续规划

## 1. 项目定位

本项目当前定位为“边缘计算与物联网融合的智能窗帘机器人装置”的多端工程基线，已经包含 STM32 嵌入式固件、Python 边缘控制程序、微信小程序、PCB/EDA 资料和 3D 结构资料。当前阶段应先把各端现状同步清楚，再推进协议、联调和功能升级。

当前项目不是单一 Python 程序，也不是只有硬件资料的归档。后续开发建议按“手机/小程序交互 -> Python 边缘控制层 -> STM32 执行层 -> 电机与机械结构”的链路逐步闭环。

## 2. 当前目录结构概览

- `01_软件/Python控制程序/`：Python 上位机/边缘控制程序，包含 `curtain_robot.py`、`motor_controller.py`、`serial_communicator.py`、`auto_controller.py`、`light_sensor.py`、`test_curtain.py`、`requirements.txt`。

- `02_硬件/PCB_AD工程/AD_Project/`：Altium Designer 相关 PCB/原理图/STEP/PDF/打板压缩包资料，包括 `Curtain_robot.PcbDoc`、`Curtain_robot.SchDoc`、`My_Curtain_robot.pdf`、`PW02.STEP`。

## 3. 已实现内容

### 3.1 STM32 嵌入式固件

状态：部分实现。

依据目录：`01_软件/STM32嵌入式固件/My_curtain_robotV2_VSCode/`。

已具备的内容：

- 工程形态已经迁移/整理为 VSCode + STM32Cube/CMake 工程，依据为 `CMakeLists.txt`、`CMakePresets.json`、`cmake/gcc-arm-none-eabi.cmake`、`.vscode/tasks.json`。
- MCU 配置为 STM32L051K8Ux / STM32L0，依据为 `My_curtain_robotV2.ioc` 中的 `Mcu.UserName=STM32L051K8Ux`、`ProjectManager.TargetToolchain=CMake`。
- 已配置 `USART1`、`USART2`、`DMA`、`I2C1`、`RTC`、`ADC` 等外设，依据为 `My_curtain_robotV2.ioc` 和 `Src/main.c` 中的 `MX_USART1_UART_Init()`、`MX_USART2_UART_Init()`、`MX_I2C1_Init()`、`MX_RTC_Init()`、`MX_ADC_Init()`。
- `USART2` 当前为 9600 波特率并使用 DMA + IDLE 接收框架，依据为 `Src/usart.c`、`Src/stm32l0xx_it.c` 中的 `HAL_UART_Receive_DMA()`、`UART_FLAG_IDLE`、`DMA1_Channel5`。
- `Src/main.c` 中已处理 `ON`、`OFF`、`C`、`S...T` 这几类串口命令：`ON/OFF` 驱动电机正反转，`C` 用于记录一次运行时间 `worktime`，`S...T` 走时间参数解析和 `Calibration_Times()`。
- 已有低功耗/唤醒相关代码，依据为 `Drivers/wakeup/wakeup.c`、`Src/main.c` 中的 `SystemPower_Config()`、`HAL_PWR_EnterSTOPMode()`、`SystemClockConfig_STOP()`。
- 已有 OPT3001 光照传感器驱动文件 `Drivers/OPT3001/opt3001.c`、`Drivers/OPT3001/opt3001.h`。

不应夸大为已完成的内容：

- 当前固件命令仍偏早期，尚未形成稳定的 `OPEN/CLOSE/STOP/STATUS/SET:x` 统一协议。
- `worktime` 主要是运行期变量，尚未形成可靠的断电保存和位置状态恢复机制。
- 已有构建产物只能说明本地曾经构建过，不能替代重新构建和硬件实测验收。

### 3.2 Python 边缘控制程序

状态：部分实现。

依据目录：`01_软件/Python控制程序/`。

已具备的内容：

- `curtain_robot.py` 已经形成主程序入口和交互式调试台，支持 `open`、`close`、`cal_open`、`cal_close`、`status`、`enable_auto`、`disable_auto` 等命令。
- `config.py` 使用 `.env`/环境变量配置控制模式、串口、GPIO、自动控制、日志和数据文件；当前默认 `CONTROL_MODE=serial`，即 Python 通过串口把命令交给 STM32 执行。
- `serial_communicator.py` 已封装串口连接、接收线程、回调、`ON`、`OFF`、`C` 命令发送，依赖 `pyserial`。
- `motor_controller.py` 已封装打开、关闭、校准开始/结束、校准时间保存与读取；校准数据写入 `curtain_data.json`。
- `auto_controller.py` 已有基于 `schedule` 的定时打开/关闭和光照变化回调框架。
- `light_sensor.py` 保留树莓派 GPIO/光线传感器备用路径，但当前读数逻辑属于简化模拟，需要真实 ADC 或传感器接入后再验证。
- `test_curtain.py` 提供不依赖实际硬件的模块级模拟测试，覆盖 GPIO、串口、电机控制、串口校准协议、光线传感器、自动控制器。
- `requirements.txt` 明确依赖 `pyserial`、`RPi.GPIO`、`schedule`、`python-dotenv`。

不应夸大为已完成的内容：

- Python 侧还没有 HTTP/MQTT API 服务。
- 自动控制虽然有框架，但默认配置中 `AUTO_CONTROL_ENABLED=False`，并且还没有完成真实硬件闭环验证。
- 位置百分比控制、完整状态机、异常保护策略、设备在线状态同步仍未完成。

## 4. 当前未实现或不完整内容

- 三端统一协议未完成：STM32、Python、小程序当前存在 `ON/OFF/C/S...T/CE` 等命令混用，尚未形成明确的命令表、返回值、错误码和状态查询协议。
- 位置百分比控制未完成：已有校准时间概念，但没有完整实现 `0%-100%` 位置模型、当前位置持久化、断电恢复和误差修正。
- 立即停止/异常保护不足：Python 的 `stop()` 只覆盖 GPIO 路径，串口模式没有稳定的 `STOP` 命令；STM32 侧也没有形成统一超时、堵转、限位、急停策略。
- 自动控制未闭环：Python 有定时/光照框架，小程序有时间选择控件，STM32 有 RTC/OPT3001 相关代码，但三者尚未联调成可验收的自动控制功能。
- 状态反馈不完整：缺少统一的 `STATUS` 返回，不能稳定展示当前位置、电机运行状态、自动模式、光照值、错误状态。
- 联网能力未实现：当前未看到 HTTP/MQTT 服务或云端接入代码，小程序主要走 BLE 原型。
- 硬件闭环证据不足：PCB、3D、固件、Python、小程序均有资料或代码，但缺少整机测试记录、接线表、BOM、演示视频、测试数据。

## 5. 当前主要风险

- 协议风险：命令命名和语义不统一，会导致后续每端各写各的，联调成本上升。
- 状态风险：缺少中心化状态机，容易出现手动控制、自动控制、校准流程互相覆盖。
- 持久化风险：STM32 侧 `worktime` 仍偏运行期记录；Python 侧只保存打开/关闭时间，尚未保存完整位置和设备状态。
- 安全风险：没有明确的堵转、超时、限位、急停和电机保护验收项。
- 硬件一致性风险：PCB/3D 资料与当前固件引脚、实际装配之间还需要逐项核对。
- 验证风险：已有测试主要是 Python 模拟测试和历史构建产物，不能替代 STM32 重新构建、实物联调和整机长时间测试。

## 6. 推荐后续开发路线

### 阶段 1：文档同步与基线冻结

- 完成本文件，明确当前实现、未实现和风险。
- 补充三端协议草案、引脚表、硬件连接表和测试清单。
- 后续所有功能分支以本文件为基准，避免把规划误认为已完成。

验收标准：文档能说明每个模块当前状态，且没有改动代码、硬件文件或工程配置。

### 阶段 2：STM32 与 Python 串口闭环

- 统一基础命令：建议至少包含 `OPEN`、`CLOSE`、`STOP`、`STATUS`、`CAL_START`、`CAL_END` 或兼容现有 `ON/OFF/C` 的过渡协议。
- Python 串口模式补齐停止、状态查询、错误处理和超时重试。
- STM32 侧返回明确状态和错误信息。

验收标准：Python 通过串口能稳定控制 STM32 打开、关闭、停止，并读取状态。

### 阶段 3：位置控制与校准持久化

- 建立窗帘位置模型，明确全开/全关校准流程。
- 支持移动到目标百分比位置。
- 保存校准时间、当前位置、最近动作和异常状态。

验收标准：断电重启后仍可基于校准数据执行可预测的位置控制。

### 阶段 4：边缘自动控制

- 在 Python 层实现稳定状态机，协调手动控制、定时控制、光照控制。
- 确认光照数据来源：STM32 OPT3001、树莓派 GPIO/ADC，或其他传感器路径只能保留一种主线。
- 增加日志和测试记录。

验收标准：不依赖云端时，设备也能按时间和光照规则自动运行，且动作原因可追踪。

### 阶段 5：小程序接入

- 优先打通 BLE 控制和状态反馈。
- 小程序 UI 从蓝牙调试页升级为用户控制页：状态、位置、自动模式、参数配置。
- 若需要远程控制，再评估 Python HTTP API 或 MQTT。

验收标准：手机端能完成连接、控制、状态查看和参数设置。

### 阶段 6：硬件与结构整机验证

- 核对 PCB 引脚、电源、电机驱动、传感器接口与固件配置。
- 验证 3D 外壳、电机、滚轮、窗帘安装方式。
- 做连续运行测试，记录发热、卡顿、噪声、松动和电源稳定性。

验收标准：整机能多次连续开合，异常阻力下能保护电机和窗帘。

### 阶段 7：展示与交付材料

- 整理架构图、接线图、协议表、BOM、测试表、演示视频和运行日志。
- 将 README、项目报告和演示材料对齐到真实实现状态。

验收标准：其他开发者可以按文档打开、构建、运行和复现实验。
