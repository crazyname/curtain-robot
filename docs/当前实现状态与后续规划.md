# 智能窗帘项目当前实现状态与后续规划

## 1. 项目定位

本项目是“边缘计算与物联网融合的智能窗帘机器人装置”的公开多模块开发基线，包含 STM32 嵌入式固件、Python 控制程序、微信小程序原型、PCB/EDA 设计资料和机械结构资料。当前目标是准确说明已有实现边界，并在此基础上推进可复核的串口闭环和实物联调。

## 2. 当前公开结构

- `01_软件/Python控制程序/`：Python 主程序、串口/GPIO 控制、自动控制与测试入口。
- `01_软件/STM32嵌入式固件/My_curtain_robotV2_VSCode/`：STM32 工程源码、驱动与 CMake/VS Code/MDK 工程配置。
- `01_软件/微信小程序/My_Curtain_robot_Wechart/`：BLE 控制原型工程。
- `02_硬件/PCB_AD工程/`、`02_硬件/立创EDA工程/`：电路与 PCB/EDA 设计资料。
- `02_硬件/机械结构_3D外壳/3D外壳设计/01_SolidWorks源文件/`、`02_STL打印文件/`：机械设计和打印资料。
- `docs/`：状态说明、串口协议草案与 BOM 初稿。
- `.github/`、`README.md`、`ROADMAP.md`、`CHANGELOG.md`、`CONTRIBUTING.md`、`LICENSE`：公开协作与入口文件。

本说明仅描述公开跟踪结构；本地备份、报告、旧副本和工具文件不是当前公开主线的组成部分。

## 3. 软件与固件现状

### 3.1 STM32 固件

已核实的基础能力：

- 工程包含电机控制、RTC、ADC、I2C/OPT3001、串口通信与低功耗相关代码基础。
- 固件串口路径已基础处理 `ON`、`OFF`、`C`、`S...T`、`STOP` 与 `STATUS`。
- `STOP` 已具备基础停止动作；`STATUS` 已能返回最小文本状态，形如 `STATE:<state>;CAL:0`。

仍需保持的边界：

- `CAL:0` 是基础占位，不代表校准结果持久化或完整校准状态已完成。
- `worktime` 与位置状态仍缺少可验证的断电保存、恢复和位置闭环。
- 超时、堵转、限位、错误码和整机安全保护仍未形成已验收方案。

### 3.2 Python 控制程序

已核实的基础能力：

- 现有程序包含主入口、串口通信、GPIO 控制、电机控制、自动控制与测试代码。
- 串口通信当前可确认发送 `ON`、`OFF`、`C` 等现有命令。

仍需保持的边界：

- Python 中显示的本地 `status` 不等同于向 STM32 发出 `STATUS` 查询。
- 串口模式尚未完成 `STOP` 发送、`STATUS` 查询和 `STATE:<state>;CAL:0` 解析闭环。
- 自动策略、传感器与真实硬件运行仍缺少整机联调证据。

### 3.3 微信小程序

- 工程已包含 BLE 搜索、连接与写入控制入口。
- 现有按钮逻辑仍写入 `ON`、`OFF`、`CE` 等旧命令形式。
- 小程序尚未接入统一 `STOP` / `STATUS` 语义或状态展示闭环。

## 4. 硬件与资料现状

- 仓库包含 PCB/EDA、SolidWorks 源文件和 STL 打印资料，可作为设计核对输入。
- [硬件物料清单（BOM）初稿](bom.md) 已存在，用于整理可从公开设计资料追溯的物料与待确认事项。
- BOM、PCB、机械结构文件均不证明样机已完成装配或通过负载、安全、续航和长期运行验证。
- 公开主线不再纳入一张未见公开再发布授权依据且带第三方水印的参考图片；这不等于历史提交已经清除，也不等于其他 CAD/STL 或第三方资源已经完成授权核验。

## 5. 主要风险与限制

- 协议与集成风险：STM32 的基础 `STOP` / `STATUS` 尚未由 Python 和小程序接入，完整跨端闭环不存在。
- 状态与持久化风险：最小状态返回和校准占位尚不足以支撑可靠的位置控制和断电恢复。
- 硬件验证风险：电源、驱动、电机负载、机械安装与停止保护需要实物验证。
- 资产合规风险：设计资料与第三方资源仍需按来源和许可逐项核验，不能整体宣称可自由再发布。

## 6. 后续路线

### 当前基线收口

- 统一 README、路线图、版本记录与本状态文档对公开结构和能力边界的描述。
- 删除已确认存在公开再发布风险的参考图片。
- 保持协议文档与 BOM 仅陈述可追溯事实和待确认事项。

### 下一优先功能任务：Python 接入 `STOP` / `STATUS`

- 在 Python 串口客户端中发送固件已有的 `STOP` 与 `STATUS`。
- 解析 STM32 当前最小状态返回 `STATE:<state>;CAL:0`，并明确异常/超时处理。
- 形成可复核的测试或最小联调记录，不把尚未验证的状态扩展写成已完成。

### 后续任务

- 完善校准状态、持久化与位置控制语义。
- 在基础串口闭环稳定后推进小程序命令统一与状态展示。
- 核对硬件版本、接线、装配和实物验收证据。

## 7. 当前结论

当前公开仓库已具备多模块设计资料和 STM32 基础 `STOP` / `STATUS` 能力，但 Python、小程序与实物整机闭环尚未完成。公开说明必须持续区分“已有代码或资料”“已接入能力”和“经过实物验证的结果”。
