# 项目路线图

本文档用于把后续开发拆分为可由协作者认领的工作项。任务被合并并有相应验证证据前，均视为未完成；认领任务时建议先创建单一范围的 Issue，再提交对应 Pull Request。

## 当前阶段

项目协作基线已经建立，仓库具备公开协作所需的入口文档、许可、贡献说明、版本记录和 GitHub 模板。STM32 固件已基础实现 `STOP` / `STATUS` 能力，为后续串口闭环提供底层起点。

目前 Python 串口客户端、小程序协议接入和真实整机控制闭环尚未完成。位置控制、状态持久化、硬件复现与自动化质量检查仍需按任务逐步验证，不能按成品能力对外描述。

## 已完成基线

- [x] `docs/当前实现状态与后续规划.md` 已提供当前实现状态文档。

## 近期任务：协议与底层控制

- [ ] **Python 串口客户端支持 `STOP` / `STATUS`**
  - 目标：在串口控制路径中发送停止和状态查询命令，并解析 STM32 当前可返回的状态结果。
  - 依赖：STM32 `STOP` / `STATUS` 基础能力；串口协议草案。
  - 验收方式：使用 Python 测试或最小联调记录证明串口模式可停止运动、查询状态，并覆盖超时或异常响应。
  - 建议 Issue 标签：`python`、`protocol`、`priority:high`

- [ ] **STM32 状态与校准信息完善**
  - 目标：明确 `STATUS` 的最小稳定字段，并补齐校准状态、运行状态及错误状态的返回规则。
  - 依赖：现有 STM32 `STATUS` 基础实现；串口协议草案中的目标返回格式。
  - 验收方式：更新协议说明并提供串口输出示例或实物联调记录，证明各字段在空闲、运行和校准场景下可解释。
  - 建议 Issue 标签：`stm32`、`protocol`、`calibration`

- [ ] **STM32 `worktime` 持久化**
  - 目标：将校准得到的运行时间可靠保存，并定义断电重启后的读取、无效值和重校准行为。
  - 依赖：校准语义确认；可用的非易失存储方案。
  - 验收方式：完成保存与恢复测试，证明重启后有效数据可读取、无效数据不会驱动错误位置推算。
  - 建议 Issue 标签：`stm32`、`persistence`、`calibration`

- [ ] **`SET:x` 位置控制**
  - 目标：基于校准行程支持 `0` 到 `100` 的目标位置控制，并约束越界参数和运动中断行为。
  - 依赖：`worktime` 持久化；稳定的状态返回；`STOP` 行为已验证。
  - 验收方式：提供多个目标位置的实物或可复核联调记录，并验证越界参数、停止和重复指令处理。
  - 建议 Issue 标签：`stm32`、`protocol`、`motion-control`

## 中期任务：边缘控制与移动端接入

- [ ] **Python 状态管理**
  - 目标：在边缘控制层维护设备连接、运动、位置估算、校准和错误状态，供自动策略及上层接口统一读取。
  - 依赖：Python `STOP` / `STATUS` 客户端能力；稳定状态字段。
  - 验收方式：增加状态转换测试，验证手动动作、状态查询、停止和串口异常不会产生矛盾状态。
  - 建议 Issue 标签：`python`、`state-management`、`testing`

- [ ] **小程序协议统一**
  - 目标：将小程序控制命令及状态展示对齐统一协议，移除或明确旧命令的兼容边界。
  - 依赖：协议字段定稿；Python 或 STM32 的可用通信入口选择完成。
  - 验收方式：使用微信开发者工具与联调记录验证开、关、停、状态查询以及错误提示的端到端行为。
  - 建议 Issue 标签：`miniprogram`、`protocol`、`integration`

- [ ] **基础控制闭环联调**
  - 目标：完成 Python 到 STM32 再到电机执行与状态返回的最小闭环，并形成可复核记录。
  - 依赖：Python 串口客户端；状态管理；接线和烧录说明初稿。
  - 验收方式：记录一次 `OPEN` / `CLOSE` / `STOP` / `STATUS` 流程的设备、步骤、响应与结果。
  - 建议 Issue 标签：`integration`、`hardware-test`、`priority:high`

## 硬件与复现任务

- [ ] **新增 `docs/bom.md`**
  - 目标：整理可采购的核心器件、数量、关键规格和替代注意事项。
  - 依赖：核对当前 PCB、执行机构、传感器和电源方案。
  - 验收方式：协作者可按清单识别搭建最小样机所需部件，未确定器件被明确标注。
  - 建议 Issue 标签：`documentation`、`hardware`、`reproducibility`

- [ ] **新增 `docs/wiring.md`**
  - 目标：给出控制板、电机驱动、传感器、电源与通信模块的接线和引脚映射。
  - 依赖：确认固件引脚配置与实际硬件版本。
  - 验收方式：接线表与图示可逐项对照，并经一次样机通电前复核或联调验证。
  - 建议 Issue 标签：`documentation`、`hardware`、`wiring`

- [ ] **新增 `docs/flashing-stm32.md`**
  - 目标：说明 STM32 工程构建、烧录工具、目标板连接方式和常见故障排查。
  - 依赖：确认维护中的 STM32 工程与可复现烧录流程。
  - 验收方式：新的协作者按文档能够完成一次固件烧录并确认基础串口响应。
  - 建议 Issue 标签：`documentation`、`stm32`、`onboarding`

- [ ] **新增 `docs/mechanical-assembly.md`**
  - 目标：描述结构件选择、打印件用途、装配顺序、固定方式及机械安全注意事项。
  - 依赖：核对现有 SolidWorks/STL 资料与实际可装配版本。
  - 验收方式：装配步骤带有部件对应关系，并记录一次样机装配检查结果。
  - 建议 Issue 标签：`documentation`、`mechanical`、`hardware`

- [ ] **新增 `docs/test-checklist.md`**
  - 目标：形成从烧录、接线、校准、基本运动到停止保护和闭环验证的检查表。
  - 依赖：基础控制闭环；BOM、接线与烧录文档。
  - 验收方式：使用检查表执行一次测试，填写结果、异常和复测结论。
  - 建议 Issue 标签：`documentation`、`testing`、`hardware-test`

  - 验收方式：另一位协作者能根据文档复现演示流程，并定位对应硬件、固件和软件版本。
  - 建议 Issue 标签：`documentation`、`release`、`hardware-test`

## Issue 拆分建议

- 每个勾选项单独建立 Issue，标题使用“模块：可验收目标”的形式，例如“Python：支持串口 `STOP` / `STATUS`”。
- Issue 描述应保留目标、依赖和验收方式，并补充相关文件范围与是否需要实物硬件。
- 同一 Pull Request 只关闭一个主要任务；协议变更涉及多端时，先合并协议约定，再分别认领实现与联调任务。
- 涉及硬件验证的任务应上传可复核的接线、步骤、环境、结果或故障记录，避免仅以功能描述作为完成依据。
